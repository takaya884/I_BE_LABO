# I.BE LABO 在庫管理システム 構想書

## 1. 背景と目的

I.BE LABO（名古屋）は、世界各国のヴィンテージコインからリング・ネックレス・ブレスレットを手作りする予約制ワークショップ。本システムは、現店舗の在庫管理を効率化しつつ、**フランチャイズ展開を前提に「店舗ごと」で在庫を独立管理できる**マルチテナント型 SaaS として設計する。

### 解決したい課題（仮説）

- ヴィンテージコインは**一点物に近い性質**（国・額面・年号・状態）で、Excel 管理では追跡が難しい
- 体験予約に対し「使えるコインがいくつあるか」がすぐ分からない
- 部材（チェーン・ケース・コーティング剤等）の発注タイミングを属人的に判断している
- フランチャイズ展開時、本部が**全店舗の在庫・売上を横断的に把握**したい

---

## 2. システムのゴール


| 観点   | ゴール                             |
| ---- | ------------------------------- |
| 現場運用 | コイン1枚単位 / 消耗品ロット単位で正確に在庫が分かる    |
| 予約連動 | 体験予約に対し必要部材が確保できているか即判断できる      |
| 仕入   | 在庫切れ・過剰在庫を防ぎ、発注タイミングを通知         |
| FC展開 | 店舗ごとに独立した在庫を持ちつつ、本部が横断集計できる     |
| 拡張性  | 将来の EC・卸・他業態（コイン以外の手作り体験）にも転用可能 |


---

## 3. ビジネス上の特性（設計に効く前提）


| 特性                                 | 設計への影響                      |
| ---------------------------------- | --------------------------- |
| コインは**個体管理**が必要（同じ国でも年号・状態で価値が違う）  | SKU だけでなく「個品 ID」を持つ二層構造     |
| 消耗品（スチールウール、コーティング剤等）は**ロット/数量管理** | 同じテーブルで両立できるよう柔軟な型を持つ       |
| ワークショップで**1回の体験 = 複数アイテム消費**       | 「体験メニュー → BOM（部品構成表）」を持つ    |
| 完全予約制                              | 予約時点で在庫を**仮押さえ**できる仕組みが望ましい |
| 店舗間で人気コインを**移動**することがある            | 店舗間移動（振替）機能                 |


---

## 4. 主要機能（MVP → 拡張）

### Phase 1（MVP）— 単店舗で回せる最小構成

- 商品マスタ管理（コイン / 部材 / 工具）
- 個体管理されるコインの登録・写真添付・状態メモ
- 入庫・出庫・棚卸の記録
- 体験メニューごとの BOM 定義
- 体験実施 1 件分の在庫消費を 1 アクションで処理
- 在庫一覧 / 在庫切れアラート
- スマホで使えるレスポンシブ UI（店頭で iPad / iPhone 利用想定）

### Phase 2 — フランチャイズ対応

- テナント（本部）／店舗の階層管理
- 店舗ごとに独立した在庫スコープ
- 役割別権限（本部管理者 / 店長 / スタッフ）
- 店舗間在庫移動
- 本部ダッシュボード（全店横断の売上・在庫・回転率）

### Phase 3 — 周辺連携

- 予約システム連携（在庫の自動仮押さえ）
- 仕入先マスタ・発注書発行
- レシート / 売上レポート（会計ソフト連携）
- EC 在庫共有（実店舗とネット販売の二重売り防止）

## 5. データモデル（ドラフト）

```
Tenant (本部・FC契約単位)
  └─ Store (店舗)
       ├─ User (スタッフ・店長)
       ├─ Item (商品マスタ: コイン種別・部材種別)
       │    ├─ ItemUnit (個体: 1枚のコイン = 1レコード)  ※個体管理品のみ
       │    └─ Stock (在庫数量・ロット)                   ※数量管理品のみ
       ├─ ExperienceMenu (体験メニュー)
       │    └─ BOM (体験1回あたりの消費アイテム)
       ├─ StockMovement (入出庫履歴: 入庫/出庫/棚卸/移動/体験消費)
       └─ Reservation (予約 → 在庫仮押さえ)
```

### 補足

- **Item に `tracking_type` フィールド**（`individual` / `quantity`）を持たせ、コインと消耗品を同じスキーマで扱う
- **StockMovement は append-only**（履歴を消さない）。在庫数は集計で算出
- **マルチテナント方式は「Row-Level Security ＋ store_id スコープ」**を基本とする（後述）

---

## 6. マルチテナント設計の方針


| 方式                        | メリット              | デメリット          | 採否        |
| ------------------------- | ----------------- | -------------- | --------- |
| 全店舗 1 DB ＋ `store_id` で分離 | 開発・運用シンプル、本部集計が容易 | 障害影響が全店に波及     | ◎ MVPで採用  |
| テナントごとに DB / スキーマ分離       | 強い分離、規制対応に強い      | 運用コスト高、横断集計が面倒 | △ 大規模時に検討 |
| サーバー自体を分ける                | 完全分離              | コスト・運用が破綻      | × 不採用     |


**MVPでは「シングル DB + `tenant_id` / `store_id` スコープ + Row-Level Security」で構築**し、規模拡大時に再評価する。

## 7. 技術スタック候補


| レイヤ    | 候補                                          | 備考                      |
| ------ | ------------------------------------------- | ----------------------- |
| フロント   | Next.js (App Router) + TypeScript           | スマホ／PC 両対応、PWA 化も視野     |
| UI     | Tailwind CSS + shadcn/ui                    | 開発スピード重視                |
| バックエンド | Next.js API Routes / もしくは Hono              | MVPはモノリスで十分             |
| DB     | PostgreSQL (Supabase)                       | RLS でテナント分離、認証・ストレージも統合 |
| 認証     | Supabase Auth（メール / Google）                 | 店舗スタッフ追加が容易             |
| 画像     | Supabase Storage                            | コイン写真の保存                |
| ホスティング | Vercel                                      | Next.js と相性◎            |
| 通知     | Resend or SendGrid（メール）／ LINE Messaging API | 在庫アラート・発注通知             |


> **判断軸**：少人数運用 × FC 展開 × スマホ操作中心のため「サーバーレス + マネージド DB」を選定。

## 8. 画面イメージ（ラフ）

- **店舗ダッシュボード**：今日の予約 / 在庫切れ警告 / 直近の入出庫
- **コイン一覧**：国 × 額面 × 年号で絞り込み、写真サムネで一覧
- **コイン詳細**：写真・状態・取得日・原価・販売実績
- **入出庫登録**：QR / バーコード対応も将来検討
- **体験消費**：「体験メニュー選択 → 一括で BOM 分の在庫を引く」1タップ
- **本部ダッシュボード（Phase 2）**：全店在庫マップ・売上 KPI

## 9. 想定ユーザーと権限


| ロール                  | 権限                        |
| -------------------- | ------------------------- |
| 本部管理者 (Tenant Admin) | 全店舗の在庫・売上閲覧、店舗追加、メニュー雛形配布 |
| 店長 (Store Manager)   | 自店舗の全権、スタッフ追加、発注          |
| スタッフ (Staff)         | 入出庫・体験消費・棚卸の登録            |
| （将来）参照のみ             | 経理・会計事務所等の外部閲覧用           |

## 10. 想定リスクと対応


| リスク                      | 対応                                |
| ------------------------ | --------------------------------- |
| コイン写真が大量になりストレージ肥大       | 自動リサイズ、CDN キャッシュ                  |
| FC オーナーが本部に在庫を見られたくないケース | 共有範囲を契約単位で設定可能に                   |
| スタッフ離職時のアカウント残留          | 招待制 + 無効化フロー                      |
| 棚卸ズレ                     | 履歴ベースで差分検出、原因タグ付け（破損 / 紛失 / 入力ミス） |

## 12. 次に決めたいこと（オープンな論点）

ここを詰めると設計が一気に固まります。

1. **コインは1枚ずつ個体管理したいか、種類＋枚数で十分か？**
  → 価格帯が均一なら数量管理で軽くできる。ヴィンテージ性を売りにするなら個体管理推奨。
2. **既存の予約システムは何を使っているか？**（連携可否を決める）
3. **POS / レジ・会計ソフトは何を使っているか？**
4. **想定 FC 店舗数は？**（10店規模か、100店規模かで設計方針が変わる）
5. **発注は本部一括か、店舗ごとに直接か？**
6. **予算感と公開希望時期**
7. **コインの仕入は海外バイヤーから？**（為替・関税の扱いが必要か）

## 13. 参考：類似プロダクトとの差別化

汎用在庫 SaaS（スマレジ・ZAICO 等）と比べた時の本システムの強みは：

- **「体験メニュー → BOM → 一括消費」**のワークフローを最初から組み込む
- **個体管理（コイン）＋ 数量管理（消耗品）の両立**
- **FC 本部 / 店舗の階層を前提**にした権限・集計
- ハンドメイド体験業態に特化した UX
